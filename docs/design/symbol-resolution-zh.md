# 符号解析

## 问题背景

在 protobuf 中解析符号比看起来更复杂。像 `Address` 这样的裸名可能指向兄弟消息、父作用域中的嵌套类型，或从另一个文件（可能经由传递依赖）导入的类型。像 `google.protobuf.Timestamp` 这样的带点号的名字可能是绝对路径，也可能以当前包为根的相对路径。语言规范要求特定的解析顺序——先向上走封闭作用域链，再搜索导入——并引入了 `public import` 进行传递重导出，在树遍历之上叠加了图遍历。

与此同时，参与解析的*文件集*取决于项目布局：Gradle 把 proto 放在 `src/main/proto/`、Bazel 从 BUILD 文件计算根、用户可能配置任意路径。IDE 插件必须处理所有这些情况而不硬编码任何一种。

挑战是三方面的：

1. **作用域嵌套** — 名字相对于任意深度的消息/枚举层级，解析必须向外走
2. **导入图** — 每个文件导入其他文件，public 导入是传递的，解析器必须遍历这个图而不重新进入已访问节点
3. **文件发现** — proto 文件的集合是项目相关的，必须可插拔

## 设计决策

### 决策 1：绝对 vs 相对作为独立代码路径

Protobuf 区分两种形式的类型引用：前导点号（`.google.protobuf.Timestamp`）强制从根进行绝对查找，而未限定或部分限定的名字（`Timestamp`、`inner.Msg`）触发相对解析。这两条路径有根本不同的算法：

- **绝对** — 将完全限定名与每个文件的包名 + 定义树匹配。不需要作用域遍历。
- **相对** — 先向上遍历封闭作用域链（`resolveInCurrent`），然后将符号与文件包名组合搜索导入。

分开它们避免了单一热路径中的条件分支，使每个算法的约束更容易推理。决策点是平凡的：语法解析器记录 `TypeName` 节点是否以点标记开头，引用类调用对应的解析器方法。

### 决策 2：通过组合接口建模作用域层级

作用域用两个可组合接口建模：`ProtobufScope`（有限定名并持有子项的容器）和 `ProtobufScopeItem`（存在于作用域内的任何东西）。`ProtobufMessageDefinition` 同时实现两者：它是属于其父的作用域项，也是包含自己字段和嵌套类型的作用域。

这种双重角色设计直接反映了 protobuf 语言模型。消息的 `qualifiedName()` 永远是 `parent.scope() + name`，递归计算。解析向下走时迭代作用域项并在第一个组件匹配时递归进入嵌套作用域，而 `resolveInCurrent` 通过从作用域跳到父作用域向上走。

关键细节是**虚拟作用域**（`ProtobufVirtualScope`）。`oneof` 和 `extend` 这样的结构在语法中看起来像作用域，但在 protobuf 语言中*不*创建新的命名作用域——oneof 的字段属于封闭消息。虚拟作用域标记为透明：`items()` 迭代助手直接将它们的子项解包到父作用域的项列表中。这防止 oneof 和 extend 在解析期间意外隐藏字段，同时仍保留用于格式化等其它目的的语法结构。

### 决策 3：通过公共前缀组合的相对解析

跨文件解析相对名字时，解析器必须回答：`com.example.v1` 包文件中的符号 `Foo.Bar` 是否匹配 `com.example.v2` 包导入文件中的任何内容？

解析器不是将所有定义扁平化到全局索引中，而是计算当前文件包和目标文件包之间的最长公共前缀，然后将符号附加到该前缀。如果得到的限定名落在目标文件的包命名空间内，解析器在目标文件的定义树中向下寻找剩余组件。

这种方法正确是因为 protobuf 的相对解析语义将包组件视为作用域边界。它也是高效的：不相关包的文件很快被前缀不匹配拒绝，单个文件更改不需要全局索引重建。

### 决策 4：带 Public-Import 传递性的导入图遍历

解析器使用显式栈进行深度优先图遍历。关键区别在于常规导入和 public 导入：

- **常规导入** — 符号仅对导入文件可用。这些文件从源文件推送一次到栈上，但它们的导入*不被*追踪。
- **Public 导入** — 符号被重新导出。遍历导入文件时，解析器将其 `public` 导入推入栈中以便进一步遍历。

这个单一的 `if (it.public())` 守卫实现了 protobuf 规范中的完整传递规则。Weak 导入被解析器识别但与常规导入在解析期间同等处理——它们主要用于构建系统提示，而非作用域语义。

### 决策 5：可插拔根提供者

可解析 proto 文件的集合不是固定的。不同的构建系统、模块配置和用户偏好都会影响哪些目录包含 proto 文件。根提供者系统用 `ExtensionPoint` 解决这个问题：

- `ProtobufRootProvider` 为给定的 PSI 上下文返回 `ProtobufRoot`（名称 + VirtualFile 目录）列表
- 具体提供者涵盖模块源、依赖模块、classpath 库、用户配置路径和内置已知类型
- `ProtobufRootResolver` 从所有注册提供者收集根，对其去重，然后相对于生成的目录集解析每个导入路径

像 `import "google/protobuf/timestamp.proto"` 这样的导入语句通过相对于每个收集到的根搜索相对路径来解析。这有意保持简单——复杂性在于*收集*正确的根，而非文件查找本身。

根计算是昂贵的（扫描模块依赖、JAR 内容），所以提供者扩展 `CachedProtobufRootProvider`，使用 IntelliJ 的 `CachedValuesManager` 和构建系统感知的修改追踪器缓存结果。

### 决策 6：上下文感知符号过滤器

并非每个符号在每个位置都有效。字段类型必须是消息或枚举，`extend` 目标必须是消息，扩展字段引用必须属于特定目标消息。解析器本身不内置上下文感知，而是让解析接受一个在拒绝无效匹配时使用的 `PsiElementFilter`。

这保持了解析器的通用性和可重用性，同时将验证推到调用点——每种引用类型（`ProtobufTypeNameReference`、`ProtobufFieldReference`、`ProtobufExtensionFieldReference`）提供自己的过滤器。同样的解析代码用不同过滤器为类型查找、字段查找和代码补全服务。

## 整体协作

用户输入 `Foo.Bar` 作为字段类型。引用系统构造 `ProtobufTypeNameReference` 并调用 `resolveRelatively`（无前导点）。解析器：

1. 向上遍历封闭作用域链（`resolveInCurrent`），检查每个作用域是否有包含 `Bar` 的子 `Foo`
2. 本地未找到时，尝试在包级别通过将相对名与文件包组合来查找当前文件
3. 将所有导入文件推入栈，对每个重复相对组合，传递地追踪 public 导入
4. 每一步，`fieldType` 过滤器拒绝不是消息或枚举的匹配

导入文件本身由 `ProtobufRootResolver` 发现，它询问每个注册的 `ProtobufRootProvider` 获取根，对其去重，然后相对于结果目录集解析每个导入路径。

## 核心洞察

设计分离了三个容易混淆的独立关注点：

- **在哪里找**（根提供者）— 项目相关、缓存、可扩展
- **按什么顺序找**（解析器算法）— 规范驱动、确定性：本地作用域优先，然后是带公共传递性的导入
- **什么算作匹配**（过滤器）— 上下文相关、由发起解析的引用注入

每个关注点因不同原因、不同速度变化。根提供者在添加构建系统支持时变更。解析算法仅在 protobuf 规范变更时变化。过滤器在新引用上下文引入时变化。保持分离意味着每个都可以独立演进而不干扰其他。
